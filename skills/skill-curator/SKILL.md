# skill-curator — 知見をスキルに振り分けて育てる

## このスキルの目的

このリポジトリのスキル群は**育て続ける前提**。読書メモ・PDF・Notion等で新しい知見が入るたびに、
それを再利用できる形に蒸留し、正しいスキルの `references/` に反映する。
このスキルは、これまで決めた**分類軸とゲート規則を人ではなくスキルとして実行する司書役**。

判断の正典は [../../docs/skill-map.md](../../docs/skill-map.md)。振り分け前に必ず読み、反映後に必ず更新する。

---

## ワークフロー

### Step 1: 取り込み（入力を読む）

入力形式に応じて中身を取得する。

- **PDF** … テキスト抽出して読む。
- **Notion** … `notion-link-fetch` スキルを使って本文を取得する。
- **メモ／貼り付けテキスト** … そのまま読む。

大量なら要点を落とさない範囲で把握する（要約が目的の場合は `summarize-text` と役割が違う点に注意——
ここでの目的は要約ではなく「道具化」）。

### Step 2: 知見を道具化する（最重要）

原文の要約ではなく、**効かせられる粒度**に変換する。詳細は [references/extraction.md](references/extraction.md)。
知見を次の4型のいずれかに落とす：**原則 / チェックリスト / レンズ / 手順**。

### Step 3: 分類する

`docs/skill-map.md` の分類軸に当てる。

- **主軸**：定める(`goal-design`) / 回す(`improve-loop`) / 育てる(`team-ops`)
- **対象**：個人 → `references/personal.md` ／ チーム・組織 → `references/team.md`
- **モード**：作る / 斬る（スキル内で吸収されるので、どちらの手順を厚くするかの判断に使う）

**1つの知見が複数スキル/複数referenceに分散するのは正常。** 分散させて構わない。

### Step 4: 配置先を判定する（ゲート）

原則は**既存スキルの `references/` への追記**。
新スキルを作ってよいのは次の**両方**を満たすときだけ：

1. 発動条件（トリガー語）が既存スキルと**衝突しない**
2. コアロジックが既存スキルと**別物**

どちらか一方でも欠けたら新設せず、既存スキルに追記する。迷ったら新設せず相談する。

### Step 5: 反映する

- 該当 `references/*.md` に追記する。**出典（書名・著者・該当箇所）を必ず明記**。
- 新しいトリガー語が生まれたときだけ、対象スキルの `description` に追記する。
- **`SKILL.md` 本体は薄いルーターに保つ**（概念解説を本体に書かない）。
- **既存記述との重複・矛盾は黙って上書きしない**。衝突を見つけたら、両者を示して
  「どちらを採るか／どう統合するか」をユーザーに確認してから反映する。

### Step 6: skill-map.md を更新する

反映した内容を `docs/skill-map.md` に反映：

- 対象スキルの references 欄・収録知見欄を更新
- 末尾「収録知見の索引」に、その知見がどのスキルに散ったかを追記
- 新スキルを作った場合はレジストリに新エントリを追加

### Step 7: 検証してコミット

- `scripts/validate.ps1` 相当のチェック（`name`＝フォルダ名、`description` 必須、命名規則）。
- SKILL.md 本文が肥大していないか（詳細が references に逃げているか）を確認。
- コミットは **`.cursorrules` に従い日本語**、スキル単位。プレフィックスは英語（例：`feat: goal-design に◯◯の知見を追加`）。
