# 記事タイプ別構成テンプレート

## tech-essay（技術的な気づき・学び・試してみた）

K.Nが一番書くことになりそうなタイプ。「すごい技術記事」ではなく「同じ立場のエンジニアへの共有メモ」くらいの温度感が正解。

```
① フック
   - 「こういうことで詰まった」「こういう発見をした」という文脈の提示
   - 読者に「あるある」と思わせる入り口
   NG: 「今回は〜について解説します」

② 背景・きっかけ
   - なぜこれに取り組んだか、何をしたかったか
   - 自分の作業の文脈（業務・副業・個人プロジェクト等）が入ると親近感が出る

③ 本題（詰まったこと・試したこと・気づいたこと）
   - コードや設定は必要最小限で、「なぜ」の説明を必ず添える
   - 「やってみたらこうだった」という体験ドリブンで書く

④ 結論・まとめ
   - 「結局こうなった」「この認識でOKだった」
   - 次にやること・気になっていることを添えると続きが書きやすい

⑤ Blueskyリンク（必須）
```

**長さ目安**: 1500〜3000字  
**見出し**: H2は2〜4個。H3は必要な場合のみ  
**コードブロック**: 必要な場合のみ。長すぎるなら要点のみ抜粋

---

## opinion（考察・持論・業界への意見）

「自分はこう思う」を出発点に、読者と一緒に考えるスタイル。答えを押し付けない。

```
① 問いの提示
   - 「〜ってどういうことなんだろう」「〜について最近ちょっと考えてた」
   - 問いかけ形式か、違和感の提示から入る

② 自分の立場・前提
   - この問いに対して自分がどういう立場や背景で考えているか
   - 「エンジニアとして」「3年くらいこの仕事してて感じるのは」など文脈を添える

③ 考えの展開（2〜3段階）
   - 主張 → 根拠 → 反論の予測 → 応答、くらいの流れ
   - 断言より「〜かもしれない」「〜だと思ってる」が K.N らしい

④ 締め
   - 「答えは出ていないけど〜」「引き続き考えていきたい」もOK
   - 読者への問いかけで終わると対話が生まれやすい

⑤ Blueskyリンク（必須）
```

**長さ目安**: 1000〜2000字  
**注意**: 断定しすぎない。「〜すべき」の多用は避ける

---

## life-log（日常・趣味・体験記）

キャンプ、ゲーム（FGO/TYPE-MOON/PS5）、日本史、作業環境などK.Nの好きなものを書くタイプ。  
文体が一番自由になれるカテゴリ。

```
① 出来事・体験の提示
   - 何があったか、何をしたかを軽く

② 体験の詳細
   - 良かったこと・面白かったこと・失敗したこと
   - 写真や具体的な固有名詞（ゲームタイトル、キャンプ場名など）があると読みやすい

③ 感想・気づき
   - 「こういうのが好きなんだな」「また行きたい」くらいの軽い余韻でOK
   - 無理に「学び」や「教訓」に落とさなくていい

④ Blueskyリンク（必須）
```

**長さ目安**: 800〜1500字  
**注意**: 「です・ます」と「だ・である」が混在しても気にしない。むしろ自然体でいい

---

## column（時事・業界トレンドへの短評）

ニュースや業界の話題をK.N視点でコメントするタイプ。短くてもいい。

```
① 話題の提示
   - 「〜というニュースを見た」「〜という話が出てた」
   - ソースはさらっと言及する程度でいい

② K.N視点のコメント
   - 「これってつまり〜ということだと思ってる」
   - エンジニア・UXデザイナーとしての解釈や感想

③ 締め
   - 一言で終わらせてもいい。「引き続き追っていきたい」くらいの軽さ

④ Blueskyリンク（必須）
```

**長さ目安**: 500〜1000字  
**注意**: 短くていい。無理に伸ばさない
