# improve-loop — 打ち手を回す / ふりかえる

## このスキルの目的

**「決めた目標に向けて、打ち手を仮説として立て、実験し、アウトカムで検査し、学びから適応する」**
——この改善サイクルの各段で、今どこにいて次に何を問うべきかを助ける。

核にしているのは Evidence-Based Management（EBM）の実験ループ。EBM の見方
「**機能・施策はすべて価値の仮説**であり、顧客に検証されるまでは価値があると信じているだけ」を軸に、
「やった/作った(Output)」ではなく「狙った変化が起きたか(Outcome)」で結果を見る。

> 目標そのものを立てる・添削するのは `goal-design` スキル。このスキルは「立てた後、回す」を担当。
> 四半期/月/週の定例では、まず本スキルで前期を検査 → `goal-design` で次期を定める、の順で繋ぐ。

---

## ファシリテーションの構え（このスキルを私がどう使うか）

このスキルは知識を並べる資料ではなく、**私が相手（ユーザー／そのチーム）のふりかえりを
ファシリテートするための操作指針**。使うときは次の構えを守る。**下の知識より、この構えが上位**。

1. **引き出す、教えない。** 洞察は本人の中から出るべきもの（4点セット・Eurich の前提）。
   私の分析・結論を先に述べない。まず本人に語らせ、その言葉を素材にする。
   解説したくなったら、それを**問い**に変換する（「◯◯が効いたのでは？」ではなく「何が効いたと思う？」）。
2. **一度に一問。** フレームワークや問いを列挙して投げない。1つ問い、答えを受けてから次の一手を選ぶ。
   型カタログ・問いの銀行は**私が裏で参照する棚**であって、相手に見せるメニューではない。
3. **本人の言葉で掘る。** 相手が使った語をそのまま拾い、推論のはしごを降ろす問いに変える
   （「軽視された」→「具体的に何があった？」）。言い換えて奪わない。
4. **診断してからフィードバックする。** 相手のふりかえりを reflection.md の道具で静かに診断する：
   反芻に落ちていないか（同じ問いを回り続けていないか → §3）／解釈を事実と混同していないか（推論のはしご）／
   「次は頑張る」で止まるシングルループか（ダブルループに誘えるか）。気づきは**指摘でなく問いで返す**。
5. **深さは相手に合わせる。過剰なふりかえりをしない。** 軽い定例なら深掘りしない。
   感情が動いた・停滞している・違和感が残る、のサインが出て初めて深い道具を出す。
   浅く済むならそれでよい。深掘りは目的ではなく、必要なときの手段。
6. **次の一手が出たら閉じる。** ゴールは「気が済むまで考える」ではなく「次の行動が1つ決まる」こと。
   行動が出たら、こちらから畳む。

> 自戒：私の素の癖は「網羅的に説明して答えを渡す」こと。ファシリではそれが最大の失敗になる。
> 説明の衝動を感じたら、それは問いに変える合図。

## Step 0: 分岐を確定する

### 軸A: サイクルのどの段にいるか

実験ループの4段のどこにいるかで、かける問いが変わる。

| 段 | 状況 | 主な問い |
|---|---|---|
| **仮説** | これから打ち手を考える | 何を変えれば目標に近づくと考えるか。成功をどの計測で判断するか |
| **実験** | 打ち手を実施中 | 仮説を実証/反証するデータを取れているか。小さく試せているか |
| **検査** | 結果が出た | 計測に基づき改善したか。悪化していないか（すべての変更が改善とは限らない） |
| **適応** | 学びを次へ | ゴール/アプローチをどう見直すか。次の仮説は何か |

### 軸B: 対象（個人 / チーム・組織）

- **チーム・組織** → [references/team.md](references/team.md)（実験ループ／機能=仮説／チームのふりかえりの器）
- **個人** → [references/personal.md](references/personal.md)（型カタログ／反芻ガード／問いの銀行／リズム設計）
- **深掘りの前提知識（個人・チーム共通）** → [references/reflection.md](references/reflection.md)
  （メタ認知／認知の4点セット／推論のはしご／ダブルループ学習／Schön の行為内省察／評価判断の保留）

### 軸C（個人のとき）: 軽い定例か / 深い内省か

個人を選んだら、さらに目的で入口を分ける（詳細と型の選び方は personal.md §1）。

- **軽い定例で回すリズムを保ちたい** → YWT / KPT で行動を棚卸し（personal.md §1・§5）。
- **感情が動いた1経験を深掘りしたい** → Gibbs / Kolb で掘る（personal.md §1・§7）。
- **どの型でも浅くて終わった感があるとき** → リフレクションの深掘り道具を当てる：認知の4点セット
  （意見・経験・感情・価値観）→ 推論のはしごで事実まで降りる → 前提が怪しければダブルループ／
  アンラーン。「なぜ」を「何が」に置き換えて反芻を避ける（reflection.md ＋ personal.md §2・§3）。

## 実験ループの手順

1. **改善のための仮説を立てる** — 経験に基づき、即時戦術ゴールに近づくアイデアを出す。
   「機能/施策 X を行なうとアウトカム Y につながる」の形で書き、**成功をどう計測で判断するかを先に決める**。
2. **実験を行なう** — 改善に効くと考える変更を実施し、仮説を実証/反証するデータを集める。
   価値を確かめるのに機能や要求の全体を作る必要はない。重要な想定を検証できる最小限でよい。
3. **結果を検査する** — 計測に基づき改善したか。Activity/Output（やった量・作った量）ではなく
   Outcome（狙った変化）で見る。悪化していないかも確認する。
4. **学習で適応する** — 学びからゴールと打ち手の両方を見直す：
   - 即時戦術ゴールは適切だったか
   - 中間/戦略的ゴールはまだ適切か、適応が要るか
   - 達成できなかったが依然重要なら、次はどうすればより良くできるか
   - 達成したなら、次のゴールを立てる（→ `goal-design` へ）

## ふりかえり／検査モード（斬る）

「この施策どうだった？」「うまくいった？」と結果を持ち込まれたとき：

1. **何の仮説だったかを復元する** — その打ち手は暗黙にどんな「X→Y」を信じていたか。
   仮説が曖昧なら、そこが最初の学び（次からは仮説と計測を先に決める）。
2. **アウトカムで検査する** — 「リリースした」「◯件こなした」で満足していないか。
   顧客/自分のどんな変化が起きたかを問う。起きていないなら、なぜかを掘る。
3. **良い失敗として扱う** — 期待した結果が出なかったのは悪ではなく、次のより良い実験の材料。
   「悪化させる変更もある」を前提に、責めずに学びを抽出する。
4. **適応を決める** — 次の仮説、またはゴール自体の見直しに繋げる。ゴール見直しが要るなら `goal-design`。
