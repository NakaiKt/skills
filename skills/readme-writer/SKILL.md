# readme-writer スキル

## このスキルの目的

**簡潔で、文脈ゼロで読める、単体で完結したREADME** を書く・直す。

READMEを書くとき、AIは説明的になりすぎて冗長になりがち。会話の中で出た疑問や確認事項を「補足」としてREADMEに足してしまう。このスキルはそれを防ぎ、削って研ぎ澄ます。

---

## 大原則：READMEは「初見・単発・文脈ゼロ」

READMEを読む人は、**これまでの会話も背景も知らない**。今この瞬間、READMEだけを見て判断する。だから：

- **会話のコンテキストを持ち込まない** — 「あなたが疑問に思ったこと」「さっき確認したこと」はREADMEに書かない
- **補足で膨らませない** — 質問への回答をREADMEに追記しない。回答は会話で返し、READMEには「必要な人が必要なときに読む情報」だけを残す
- **一読で伝わる構造にする** — 上から読んで詰まらない。前提説明の入れ子を避ける

READMEはドキュメントであって、議論のログでも思考の過程でもない。

## もう一つの原則：READMEは「入口」で「腐らない」

READMEは長期的に嘘をつかず、肥大化しないことも大事。

- **全部書かない** — READMEは入口。詳細・運用・設計背景が増えたら `docs/` に逃がしてリンクする
- **正は1箇所に置く** — コードや設定に既にある情報を複製しない。可変の数・バージョン・設定の全リストは、片方が変わった瞬間に嘘になる

---

## Step 0: 何をするか判定

| リクエスト | アクション |
|-----------|-----------|
| **新規作成** | 対象を把握 → 骨組み（下記の標準構成）→ 本文 |
| **添削・簡潔化** | 現状のREADMEを受け取り → アンチパターン検出 → 削って書き直し |
| **レビュー** | チェックリスト（Step 4）で診断 → 具体的な直し方を提示 |

明らかな場合は確認せず進む。

---

## Step 1: アンチパターンを検出して削る

**これが本スキルの核心。** 以下があれば削る・言い換える。

### 削る対象

| アンチパターン | なぜダメか | どうする |
|--------------|----------|---------|
| 会話由来の疑問・確認 | 初見の読者に無関係 | 丸ごと削除 |
| `> 補足:` `> 注:` の乱用 | 本文が読めれば不要 | 本文に統合 or 削除 |
| 表と本文の二重説明 | 同じ情報を2回読ませる | どちらか一方に寄せる |
| 「早見表」＋詳細の重複 | 全体量が倍になる | 表で足りるなら詳細を削る |
| 理由・背景の過剰説明 | 「なぜこうしたか」は多くの場合不要 | 手順・結論だけ残す |
| 説明的で長い一文 | 読むのに時間がかかる | 短い断定文に分ける |
| 冗長な修飾 | 「〜するためのもの」「〜という流れは同じ」 | 名詞で言い切る |
| 陳腐化する情報 | バージョン・可変の数・設定の複製は、コードが変わると嘘になる | 正は1箇所（コード/設定）に置き参照する |
| README に全部書く | 入口が肥大化して読めなくなる | 詳細・運用・背景は `docs/` へ逃がしリンク |
| ロードマップ・未実装の混入 | 「今動くもの」が判別できなくなる | 末尾か別ファイルに隔離 |
| コマンドの可変部が曖昧 | コピペで誤動作・詰まる | `<your-token>` のように可変部を明示 |

### 判断基準

各文・各セクションに問う：**「初見の人がこの1文で詰まらず前に進めるか？」「これが無いと困るか？」**
無くても困らないなら削る。

---

## Step 2: 標準構成（新規作成時）

過不足ないREADMEの骨組み。**該当しないセクションは作らない**（空セクションや無理な埋めをしない）。

```
# プロジェクト名

1〜2文で「何か」を言い切る。

## （必要なら）前提条件
動かすのに必要なランタイム/バージョン・依存・権限・OS。手順の前に示す。

## （必要なら）全体像
図やディレクトリ構成を1つ。説明は最小限。

## セットアップ / 使い方
手順を番号付きで。コマンドはコピペできる形（可変部は <...> で明示）。
最後に「こう叩けば動いたと分かる」動作確認を1つ入れる。

## （必要なら）その他
補足的な運用・注意点。ただし最小限。詳細は docs/ へ逃がす。
```

- **冒頭の1〜2文が最重要**。ここで「何のリポジトリ/ツールか」が伝わること
- **前提条件は先に**。初見の読者は前提を共有していないので、手順の前で詰まらせない
- ディレクトリ構成やコマンドは、コメントを削って要点だけ残す

---

## Step 3: 書き方ルール

| 項目 | ルール |
|------|--------|
| 文体 | 説明文ではなく断定。「〜するためのものです」→「〜するリポジトリ」 |
| 一文の長さ | 一読で頭に入る長さ。長い一文は分割 |
| 見出し | 読者の目的（「使い方」「追加する」）で切る。設計思想の解説に見出しを割かない |
| 表 | 分岐・対応関係があるときだけ使う。散文で済むなら表にしない |
| コードブロック | 実際にコピペする手順・構成図に使う。装飾に使わない |
| 重複 | 同じ情報は1箇所だけ。表と本文の両方に書かない |
| コメント注釈 | ディレクトリ構成のコメントは「役割が一目で分かる最小語」に。長い説明を足さない |

---

## Step 4: レビュー / 最終チェックリスト

```
- [ ] 冒頭1〜2文で「何か」が分かるか
- [ ] 会話・その場の疑問に由来する記述が混ざっていないか
- [ ] 「補足」「注」で膨らんでいないか（本文に統合 or 削除できないか）
- [ ] 表と本文、早見表と詳細で同じことを二度書いていないか
- [ ] 「なぜこうしたか」の背景説明が過剰でないか
- [ ] 各セクションは無いと困るか（飾りのセクションはないか）
- [ ] 一文が長すぎないか、説明的すぎないか
- [ ] コマンド・手順はそのままコピペで動く形か（可変部は明示されているか）
- [ ] 前提条件（必要な環境・依存・権限）が書かれているか
- [ ] コードや設定と重複する、腐りやすい情報を書いていないか
- [ ] README に詰め込みすぎず、詳細は別ドキュメントへ逃がしているか
- [ ] 動作確認の手順があるか
```

「削れる箇所はもうないか？」を最後にもう一度問う。**迷ったら削る。**

---

## 出力フォーマット

**新規作成：** README全文（そのまま貼れる形式）。

**添削・簡潔化：**
1. 書き直したREADME全文
2. 削った理由のサマリー（何を・なぜ削ったか、箇条書きで簡潔に）

**レビュー：** チェックリストの診断結果と、具体的な直し方（「この記述 → こう削る/こう言い換える」）。

会話の中で読者から質問を受けても、その回答をREADMEに足さない。回答は会話で返す。READMEに反映すべきか迷ったら「初見の人に必要か」で判断し、必要なときだけ最小限で入れる。
