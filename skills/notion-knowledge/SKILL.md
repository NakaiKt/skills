## ワークフロー

### 1. ナレッジを整理する

1. **ナレッジ名**: 「[ツール名] [動詞] [対象]」形式が検索しやすい  
   例: `npm permissionエラーの解消`, `useCallbackの依存配列の正しい設計`  
   抽象的な名前（`Rubyメモ`, `SQLまとめ`）は後から検索しても中身が思い出せない。**一歩踏み込んだ具体的な技術名**を入れる（`便利メソッド`ではなく`freezeで文字列を不変にする方法`のように）。エラー系はエラーメッセージそのものを含めると再検索でヒットしやすい
2. **概要**: 3文以内で問題・解決策・注意点を要約
3. **テーマ・ツール**: 最大2〜3個に絞る。既存にない場合は新規追加OK
4. **ステータス**: 作業済みなら `完了`、後で肉付けするなら `未着手`
5. **アイコン・カバー**: 上記マッピングから決定する

### 2. レコードを作成する

`notion-create-pages` で作成。`icon` と `cover` を必ず含める：

```
parent: { data_source_id: "314c77a9-8998-80fa-b6e0-000b649f4438" }
pages:
  - properties:
      ナレッジ名: "<タイトル>"
      概要: "<要約テキスト>"
      テーマ: '["<テーマ1>"]'
      ツール: '["<ツール1>", "<ツール2>"]'
      ステータス: "完了"
    icon: "<絵文字>"
    cover: "<カバーURL>"
    content: |
      <本文>
```

### 3. ページ本文のフォーマット

```markdown
## 背景・問題
何が起きたか、何をしたかったか

## 解決策・手順
具体的なコードやコマンドを含む

## 動作確認環境
- 検証した言語・ライブラリ・OS等のバージョン（例: Ruby 3.4.6 / Rails 8.0.3 / macOS 15.7）

## 注意点・ポイント
ハマりやすいポイント、関連知識

## 参考
参考にした一次情報のリンク（下記「正確性・裏取りのポイント」を参照）
```

内容によってセクションは柔軟に調整してよい。ただし **`動作確認環境` と `参考` は技術ナレッジでは極力残す**（時間が経つと陳腐化するため、「いつ・どの環境で確認したか」「どこを見れば正しい最新情報にたどり着けるか」を必ず記録しておく）。

### 4. 既存ナレッジの検索・参照

```
notion-search:
  query: "<検索キーワード>"
  data_source_url: "collection://314c77a9-8998-80fa-b6e0-000b649f4438"
```

### 5. 既存ページの更新

`notion-fetch` でページIDを取得後、`notion-update-page` で更新する。

## 正確性・裏取りのポイント

ナレッジは「未来の自分や誰かが信頼して使う情報」なので、正確さと追跡可能性を担保する。（出典: 伊藤淳一『技術記事を書く技術』第7章）

### 一次情報で裏を取る

技術情報は **一次情報**（公式ドキュメント、ライブラリのREADME、コード、GitHubのissues/PR、開発者本人のブログ）と **非一次情報**（他人のQiita/Zenn記事、個人ブログ、生成AIの回答）に分けられる。

- 非一次情報だけを根拠にナレッジを残すと、正しさが担保されない
- 生成AI（Claude / ChatGPT等）の回答も**非一次情報**。「AIの言うとおりにやったら動いた」で終わらせず、念のため一次情報で裏付けを取ってから記録する
- OSSなら、コードを読む・issues/PRを確認するのが仕様変更の理由まで含めて一番確実

### 参考文献のリンクを残す

トラブルシューティングや調査系のナレッジでは、参考にした情報源のリンクを `参考` セクションに残す。一番関連の深い本文のすぐ近くに置くスタイルでもよい。リンクがあると、後から詳細を追える・内容の怪しい部分を検証できる・正しさがある程度担保される。

### GitHub のコードを引用するとき

- **main/master ブランチではなく、タグ（バージョン）のURLを使う**。main/masterは頻繁に更新され、行番号がズレたりファイルごと消えたりする
- 行番号アンカーを付けると該当箇所へ直接ジャンプできる: 単一行 `#L214` / 範囲 `#L214-L227`
- 例: `https://github.com/minitest/minitest/blob/v5.26.2/lib/minitest/assertions.rb#L214-L227`

### 陳腐化への備え

- 参考にした記事は**投稿日・対象バージョン**を確認する（古い記事は「当時は正しかったが今は非推奨」がよくある）
- 本文に日付を書くときは**年まで含める**（「現時点」ではなく「2026年3月現在」のように書く）

### 表記を正確にする

言語・ライブラリ・サービス名の表記ゆれに注意する。確信がなければ公式サイトで確認する。

## 判断基準：何をナレッジとして残すか

- **トラブルシューティング**: エラー解決、環境構築のハマりポイント
- **設計判断**: アーキテクチャの選択理由、パターンの使い分け
- **コマンド・手順**: 再利用するシェルコマンド、セットアップ手順
- **API・ツールの使い方**: 忘れやすいオプション、挙動の注意点
- **ベストプラクティス**: コードレビューで学んだこと、リファクタリングの知見

## 注意事項

- `テーマ` と `ツール` はJSONの配列文字列として渡す: `'["React", "TypeScript"]'`
- `ステータス` に設定できる値: `未着手` / `進行中` / `完了` / `投稿`
- `最終更新日時` は自動設定されるため、プロパティに含めない
- ナレッジ名は日本語・英語どちらでもOK（検索しやすい方を選ぶ）
- カバー画像URLが表示されない場合はAI活用のURLで代替する
